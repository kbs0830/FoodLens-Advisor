# FoodLens Advisor 專題報告

**專題名稱**：FoodLens Advisor - Thin Client + Cloud Brain 的智慧飲食分析系統  
**專題類型**：AI 飲食分析 Web 應用  
**撰寫日期**：2026 年 5 月 23 日

---

## 摘要

FoodLens Advisor 是一套結合前端影像辨識與後端語意分析的智慧飲食分析系統。系統以 Web 為主要操作介面，先在瀏覽器端對使用者上傳的食物圖片進行偵測與標籤整理，再將文字化結果送至 FastAPI 後端進行營養估算、飲食規則檢查與下一餐建議生成。相較於傳統直接將圖像送入雲端 AI 的做法，本專題採用 Thin Client + Cloud Brain 的混合式架構，能有效降低 token 成本、減少圖片傳輸量，並提升隱私保護與系統回應效率。本文完整說明研究動機、相關技術、系統設計、實作架構與整體成果，並提出後續可擴充方向。

**關鍵字**：食物辨識、Web AI、FastAPI、TensorFlow.js、BFF、LLM、飲食分析

---

## 一、研究題目與動機

### 1.1 研究題目
本專題以「**結合前端食物影像偵測與後端大語言模型分析的智慧飲食分析系統**」為研究題目，目標是建立一套可在瀏覽器端完成初步辨識，再將結果交由後端進行營養分析與飲食建議的應用系統。

### 1.2 研究背景
在健康管理與飲食控制日益重要的情況下，使用者常需要快速判斷餐點是否符合飲食目標，例如高蛋白、低澱粉、避免刺激性食物等。然而，實務上常面臨以下問題：

- 使用者不易即時且準確記錄每一餐內容。
- 純手動輸入食物名稱耗時，且容易遺漏關鍵資訊。
- 若直接上傳圖片給雲端 AI 分析，成本與隱私風險會同步上升。
- 傳統規則式系統雖可控，但缺乏自然語言層次的說明能力。

因此，本專題希望設計一個能在可接受成本下運作，同時兼顧可用性、隱私與可擴充性的飲食分析系統。

### 1.3 研究動機
本研究的動機主要包含以下四點：

1. **降低雲端分析成本**  
   若將圖片直接送入多模態模型，會產生較高的 token 與推論成本。透過前端先完成食物辨識，將圖像轉為文字描述後再送往後端，可大幅縮減處理負擔。

2. **強化使用者隱私保護**  
   食物圖片可能包含個人用餐環境、地點與其他敏感資訊。若只傳送必要的文字化辨識結果，可有效降低圖片外洩與資料暴露風險。

3. **提升互動效率與可用性**  
   前端先行推論可讓使用者更快看到初步結果，不必等待完整雲端圖像分析流程，整體操作體驗更即時。

4. **展現前後端協作的 AI 架構設計能力**  
   本專題整合前端機器學習、後端 API 與語言模型分析，能作為 Web AI 應用與系統整合的實作範例。

### 1.4 研究目的
本專題的具體目的如下：

- 建立可上傳食物圖片的 Web 使用介面。
- 在前端完成食物偵測與文字摘要。
- 透過 FastAPI 後端接收文字結果並進行營養分析。
- 提供熱量估算、三大營養素與飲食規則檢查。
- 生成具有實用性的下一餐建議與飲食提醒。
- 設計可擴充的混合式架構，方便未來接入 Gemini 或其他 AI 服務。

## 二、相關研究探討

### 2.1 食物影像辨識與電腦視覺
電腦視覺在近年已廣泛應用於物件偵測、場景理解與內容分類。食物辨識屬於其中的重要應用之一，主要方法包括：

- 即時物件偵測模型，例如 YOLO 系列。
- 影像分類模型，用於辨識整張圖片中的食物種類。
- 結合區域偵測與分類器的混合式方法。

本專題採用前端偵測與分類並行的方式，先將圖片中的可疑食物候選項抽取，再交由後端進行營養與語意分析。

### 2.2 大語言模型在飲食建議中的應用
大語言模型具備自然語言生成與語意整合能力，適合用於將偵測結果轉換為可理解的飲食建議，例如：

- 判斷餐點是否符合高蛋白、零澱粉等條件。
- 根據食物種類生成下一餐建議。
- 將營養資訊整理成使用者容易理解的說明文字。

與規則式模板相比，LLM 能提供更具彈性且更自然的分析內容。

### 2.3 前端推論與薄客戶端架構
Web AI 的一項重要趨勢是將部分推論工作前移至瀏覽器端，形成 Thin Client + Cloud Brain 的架構。其優勢在於：

- 降低後端負載。
- 減少敏感圖像資料傳輸。
- 提高跨平台可用性與部署彈性。

本專題即以此為核心設計理念：前端負責辨識與篩選，後端負責整合分析與建議生成。

### 2.4 與既有飲食分析系統的差異
現有飲食分析工具多半有兩種典型模式：

- **直接上傳圖片至雲端 AI**：流程簡單，但成本高、隱私風險較大。
- **完全手動輸入飲食資訊**：成本低，但操作負擔高，且不易即時。

本專題採中間路線，透過前端先行偵測把圖片轉為文字，兼顧自動化、成本效益與使用便利性。

## 三、研究方法與系統目標

### 3.1 研究方法
本專題採取系統開發與原型驗證並行的方法，依序完成需求分析、架構設計、前端偵測模組、後端分析 API、整體串接與錯誤回退機制。

### 3.2 設計原則
整體系統設計遵循以下原則：

- **最小傳輸原則**：能在前端處理的資料，不再完整送往後端。
- **模組化原則**：前端辨識、後端分析、資料模型與 UI 顯示分層處理。
- **可回退原則**：任一模型或外部服務失效時，系統仍保有可展示結果。
- **可擴充原則**：未來能替換分類模型或 AI 供應商，而不需大幅重構。

### 3.3 系統目標
本系統希望達到下列目標：

- 提供清楚直觀的食物分析流程。
- 將圖像辨識與語意分析拆分，提高效率。
- 建立可供課堂展示與後續延伸的完整專題原型。

## 四、研究內容

### 4.1 系統功能
系統主要包含以下功能：

- 圖片上傳與即時預覽。
- 食物偵測與候選標籤整理。
- 營養成分與熱量估算。
- 飲食規則檢查，例如高蛋白、零澱粉、零酒精、清淡不辣。
- 下一餐建議與營養提醒。
- API 連線測試與錯誤回退機制。

### 4.2 系統流程
整體資料流程如下：

1. 使用者在前端上傳食物圖片。
2. 前端載入模型並執行食物偵測。
3. 將偵測到的食物名稱、信心度與描述整理為文字資料。
4. 將文字資料送往後端 `/api/v1/analyze-text`。
5. 後端根據文字內容估算營養與規則符合度。
6. 將分析結果以 JSON 回傳並顯示於前端。

### 4.3 研究成果
目前專題已完成以下成果：

- Web 前端可進行圖片上傳、預覽與結果顯示。
- FastAPI 後端已提供健康檢查與多個分析端點。
- 前端已整合食物分類與偵測回退策略。
- 後端支援 Mock 與 Gemini 等分析模式。
- 回傳結果包含營養素、飲食規則、建議與可信度提示。

## 五、系統設計與技術探討

### 5.1 整體架構
本專題採用前後端分離架構，並以 BFF（Backend for Frontend）作為中介層：

- **前端 web-client**：負責使用者介面、圖片預覽、模型推論與結果呈現。
- **後端 bff-fastapi**：負責接收請求、整合分析邏輯與回傳標準化 JSON。
- **模型與 AI 服務**：包含前端偵測模型與後端文字分析供應商。

此架構可使前端僅送出必要資訊，降低資料量與處理成本。

### 5.2 前端模組設計
前端主要由 HTML、CSS 與 Vanilla JavaScript 構成，並加入模型推論模組。

#### 5.2.1 使用者介面
前端介面包含：

- 圖片上傳區。
- 圖片預覽區。
- 偵測狀態顯示。
- 分析結果區。

#### 5.2.2 模型推論流程
依據 [web-client/yolo-detector.js](../web-client/yolo-detector.js)，前端採用食物分類模型與 COCO-SSD 回退策略：

- 優先使用食物分類模型取得較具體的類別結果。
- 若分類模型不可用，則回退至 COCO-SSD 進行一般物件偵測。
- 透過信心度門檻篩選可能的食物候選項。
- 將結果整理成 `food_items`、`confidence_scores` 與 `description`。

#### 5.2.3 前端串接流程
依據 [web-client/script.js](../web-client/script.js) 的設計，前端會依序執行：

- 呼叫 `/health` 驗證後端服務狀態。
- 呼叫 `/api/v1/detect-food` 取得偵測結果。
- 呼叫 `/api/v1/analyze-food` 或 `/api/v1/analyze-text` 取得飲食分析結果。

### 5.3 後端模組設計
後端以 FastAPI 實作，具備高效、簡潔與型別清楚的特性。

#### 5.3.1 路由設計
依據 [bff-fastapi/app/main.py](../bff-fastapi/app/main.py)，後端提供以下路由：

- `GET /health`：健康檢查。
- `POST /api/v1/analyze-food`：圖像分析相容端點。
- `POST /api/v1/detect-food`：圖片偵測端點。
- `POST /api/v1/analyze-text`：文字分析端點。

#### 5.3.2 資料模型設計
依據 [bff-fastapi/app/schemas.py](../bff-fastapi/app/schemas.py)，系統透過 Pydantic 定義請求與回應格式，包括：

- `AnalyzeFoodRequest`
- `DetectFoodRequest`
- `AnalyzeTextRequest`
- `AnalyzeFoodResponse`
- `DetectFoodResponse`
- `DetectionMeta`
- `MacroNutrients`
- `DietaryRuleCheck`

此設計可確保 API 之間的資料格式一致，降低前後端溝通與除錯成本。

#### 5.3.3 文字分析邏輯
依據 [bff-fastapi/app/services/vision_service.py](../bff-fastapi/app/services/vision_service.py)，後端支援以下分析策略：

- 若 provider 設為 `mock`，則回傳模擬分析結果。
- 若 provider 設為 `gemini`，則嘗試呼叫 Gemini 產生分析結果。
- 若食物清單為空，則回退至預設資料，避免流程中斷。

這使系統即使在模型不可用或 API 尚未配置時，仍可維持穩定展示。

### 5.4 資料流與隱私設計
本專題的一項核心設計，是將圖像辨識盡量保留在前端，讓後端接收較小的文字描述。其效益如下：

- 減少圖片傳輸量。
- 降低 token 消耗。
- 降低雲端 AI 成本。
- 降低使用者圖片外洩風險。

### 5.5 錯誤處理與回退機制
系統設計了多層回退機制，以提升穩定性：

- 偵測失敗時仍可顯示空結果。
- 食物清單為空時，後端自動採用 mock 分析。
- Gemini 失敗時回退到 mock 回應。
- 前端會顯示狀態訊息，讓使用者清楚掌握流程狀態。

### 5.6 技術選型說明
本專題選用以下技術：

- **FastAPI**：開發快速、文件清楚、適合建立 API。
- **Vanilla JavaScript**：不依賴大型框架，便於教學展示與除錯。
- **TensorFlow.js / COCO-SSD 類模型**：可在瀏覽器端進行推論。
- **Pydantic**：適合定義資料 schema 與 API 結構。
- **CORS Middleware**：方便本機前後端分離測試。

## 七、結論與未來展望

### 7.1 結論
本專題完成了一個以「前端偵測 + 後端文字分析」為核心的智慧飲食分析系統。相較於直接上傳圖片給雲端 AI 的方式，本專題在資料傳輸量、分析成本與隱私保護上都有較佳表現，同時仍能提供具實用性的飲食分析與下一餐建議。

本研究也證明，透過前端模型推論與後端語意分析的分工方式，可以有效將 Web AI 應用落實到真實場景，並兼顧系統可維護性與擴充性。

### 7.2 未來展望
未來可朝以下方向持續擴充：

- 導入更準確的食物辨識模型。
- 接入正式的 Gemini 或其他多模態 AI 供應商。
- 增加使用者登入與個人飲食偏好設定。
- 建立餐點歷史紀錄與統計分析功能。
- 加入手機相機拍攝與即時分析流程。
